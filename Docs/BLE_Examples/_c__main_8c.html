Detailed Description

This application contains an example which shows how implementing a Bluetooth LE security scenario on a central device.

KEIL project

To use the project with KEIL uVision 5 for ARM, please follow the instructions below:

  1. Open the KEIL uVision 5 for ARM and select Project->Open Project menu.
  2. Open the KEIL project C:\Users\{username}\ST\BlueNRG-LP_LPS DK x.x.x\Project\BLE_Examples\BLE_Security\MDK-ARM\{STEVAL-IDB011V1|STEVAL-IDB012V1}\BLE_Security.uvprojx
  3. Select desired configuration to build
  4. Select Project->Rebuild all target files. This will recompile and link the entire application
  5. To download the binary image, please connect an USB cable in your board (CMSIS-DAP upgrade).
  6. Select Debug->Start/Stop Debug Session to download the related binary image.
  7. Alternatively, open the Flasher utility and download the built binary image.

IAR project

To use the project with IAR Embedded Workbench for ARM, please follow the instructions below:

  1. Open the Embedded Workbench for ARM and select File->Open->Workspace menu.
  2. Open the IAR project C:\Users\{username}\ST\BlueNRG-LP_LPS DK x.x.x\Project\BLE_Examples\BLE_Security\EWARM\{STEVAL-IDB011V1|STEVAL-IDB012V1}\BLE_Security.eww
  3. Select desired configuration to build
  4. Select Project->Rebuild All. This will recompile and link the entire application
  5. To download the binary image, please connect an USB cable in your board (CMSIS-DAP upgrade).
  6. Select Project->Download and Debug to download the related binary image.
  7. Alternatively, open the Flasher utility and download the built binary image.

WiSE-Studio project

To use the project with WiSE-Studio IDE (GCC toolchain), please follow the instructions below:

  1. Open the WiSE-Studio IDE
  2. Select File, Import, Existing Projects into Workspace C:\Users\{username}\ST\BlueNRG-LP_LPS DK x.x.x\Project\BLE_Examples\BLE_Security\WiSE-Studio\{STEVAL-IDB011V1|STEVAL-IDB012V1}
  3. Select desired configuration to build
  4. Select Project->Build Project. This will recompile and link the entire application
  5. To download the binary image, please connect an USB cable in your board (CMSIS-DAP upgrade).
  6. Select Run->Run/Debug to download the related binary image.
  7. Alternatively, open the Flasher utility and download the built binary image.

Usage

This demo addresses some examples of Bluetooth Low Energy Security scenarios:

  • Pairing with pass key entry method (no fixed pin) and bonding (select: Master_PassKey_Random and Peripheral_PassKey_Random projects configurations)
  • Pairing with pass key entry method (with fixed pin) and bonding (select: Master_PassKey_Fixed and Peripheral_PassKey_Fixed projects configurations)
  • Pairing with just works method and bonding (select: Master_JustWorks and Peripheral_JustWorks projects configurations)
  • Pairing with Numeric Comparison(Bluetooth LE V4.2 security feature) and bondig (select: Master_NumericComp and Peripheral_NumericComp projects configurations)

In order to exercise each specific security scenario, each Master project configuration must be used with the related Peripheral project configuration as follows:

Central device is using the Central/Master library (refer to the related html documentation for more information). 

 Pairing with pass key entry method (no fixed pin) and bonding:

On reset, after initialization, Central device uses the Master_SecuritySet() API for setting the security IO capability as keyboard only (IO_CAP_KEYBOARD_ONLY) and the authentication requirements as follow:

  • MITM authentication (MITM_PROTECTION_REQUIRED)
  • don't use fixed pin (DONOT_USE_FIXED_PIN_FOR_PAIRING)
  • Secure connection supported and mandatory (SC_IS_MANDATORY: it forces LE Secure connections, if peripheral supports it with at least SC_IS_SUPPORTED option)
  • Key press notification not supported (KEYPRESS_IS_NOT_SUPPORTED)
  • Identity address public (0x00)
  • bonding required (BONDING)

Central device starts a device discovery procedure (Master_DeviceDiscovery() API), looking for the Peripheral device SlaveSec_A2. When found, Central connects to the Peripheral device. In order to start the pairing, Central device is expecting the Peripheral device to send a slave security request. Once the security request is received, Central device starts the pairing procedure which leads to the reception of the pass key request on Master_Pairing_CB() user callback. As consequence, Central device is asked to insert the Pass Key defined and displayed on Peripheral device, digit by digit and add it through the Central/Master Library Master_Passkey_Response() API.

NOTEs:

  • Peripheral device defines and displays the security key on related hyper terminal, since it is defined with IO capability display only.
  • Central device must provides the same security key defined and displayed on Peripheral side: each time Central device is asked to confirm the Pass key, user must press the 'y' key on Central hyper terminal. If user wants to re-enter the pass key, he must select any other key on keyboard.

Once the pairing and bonding procedure has been completed, the Central device starts the service discovery procedure in order to find the Peripheral characteristics:

  • TX characteristic: notification (CHAR_PROP_NOTIFY), no security (ATTR_PERMISSION_NONE);
  • RX characteristic with properties: read (CHAR_PROP_READ), Link must be encrypted to read (ATTR_PERMISSION_ENCRY_READ), Need authentication to read (ATTR_PERMISSION_AUTHEN_READ), GATT_NOTIFY_READ_REQ_AND_WAIT_FOR_APPL_RES (application is notified when a read request of any type is received for this attribute).

After Service Discovery, Central enables the TX characteristic notification. Then the Central device receives periodically the TX characteristic notification value from Peripheral device and read the related RX characteristic value from Peripheral device.

Central device starts a device discovery procedure (Master_DeviceDiscovery() API), looking for the Peripheral device SlaveSec_A3. When found, Central connects to the Peripheral device. In order to start the pairing, Central device is expecting the Peripheral device to send a slave security request. Once the security request is received, Central device starts the pairing procedure.

On both devices the confirm numeric value event is triggered and the related numeric value is displayed. User is requested to press key y in order to confirm the numeric value on both devices (any other key for not confirming it).

Once the pairing and bonding procedure has been completed, the Central device starts the service discovery procedure in order to find the Peripheral characteristics:

  • TX characteristic: notification (CHAR_PROP_NOTIFY), no security (ATTR_PERMISSION_NONE);

After Service Discovery, Central enables the TX characteristic notification. Then the Central device receives periodically the TX characteristic notification value from Peripheral device and read the related RX characteristic value from Peripheral device.
